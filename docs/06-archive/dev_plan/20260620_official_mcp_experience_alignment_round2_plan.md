# 对齐官方 MCP 浏览器体验收口总计划

日期：2026-06-20

## 1. 目标

这份计划不再对应单独一轮，而是把后续原本还可能拆开的几轮合并为一次性收口计划。

目标是直接把当前项目从“高质量 Beta / RC 前段”推进到“接近 1.0 的收口状态”。

重点不是继续做治理层包装，也不是继续补外围能力，而是直接提升浏览器内核交互本身的体验质量，让真实复杂页面上的操作更自然、更少探测、更少补救步骤，并把后续应当补的稳定性、性能、文档、发布标准一次性纳入同一份计划。

## 2. 目标结果

本计划完成后，浏览器核心能力应达到以下状态：

- 在复杂动态页面上，高层动作的一次命中率更高
- 在 Gmail、YouTube Studio、GitHub 这类重前端页面上，结构化提取更稳定
- 更少依赖原始 `run_script(...)` 当作最后真相面
- 动作结果、`post_action_context`、`structured_page`、下一步推理之间的连续性更强
- 默认 `patchright` 主路径的体验进一步接近官方 `playwright-mcp`
- 复杂页面上的高阶稳定性和边角场景显著收口
- 诊断输出、高噪音场景、引擎切换建议、性能基线、发布验证体系一起进入可长期维护状态

## 3. 总范围

本计划包含以下全部内容，并以“一次性收口”为目标，不再拆成后续独立主迭代。

### 3.1 结构化提取增强

- 强化复杂动态页面上的结构化提取质量，重点覆盖：
  - custom elements
  - overlay / dialog / menu / listbox
  - 重复集合项
  - 搜索栏、筛选栏、工具栏
- 优先增强“局部交互热点区域”的描述能力，而不是动不动退回全页粗扫描
- 提高以下集合场景的识别质量：
  - 邮件列表
  - 评论/线程列表
  - 仓库/文件/搜索结果列表
- 压低低价值噪音输出，让结构化结果更适合继续推理

### 3.2 高层动作连续性增强

- 强化 `post_action_context`，让它更像默认的“下一步继续操作面”
- 提升以下动作后的上下文质量：
  - `click`
  - `type`
  - `select`
  - `navigate`
  - `wait`
- 提高下一步建议的质量，建议来源包括：
  - `structured_page`
  - 当前交互区域
  - 候选项排序
  - 最近失败记录
- 降低调用方在正常多步流程里立刻补这些动作的概率：
  - `snapshot`
  - `diagnose_page`
  - 原始脚本探测

### 3.3 `run_script` 边界继续补强

- 保留并继续强化当前的 `script_result_state` 统一契约
- 增强运行时 wrapper，让常见结构化读取更容易被稳定序列化
- 更明确地区分以下几类情况：
  - 页面未准备好
  - 逻辑结果为空
  - 返回值不可序列化
  - 脚本本身没有返回值
- 在 `run_script_batch(...)` 这类路径上，继续补充更适合上游使用的批量诊断信息

### 3.4 候选解析与排序增强

- 增强以下类型目标的排序质量：
  - 主动作按钮
  - 搜索框
  - 筛选器
  - 临时弹出菜单项
  - 对话框内局部控件
- 优先做局部作用域内解析，再退回全页枚举
- 让 `match_reason` / `ranking_reason` 对调用方更有指导意义，减少盲试

### 3.5 默认主路径继续对齐官方体验

- 继续保持 `patchright` 为默认主引擎
- 继续保持：
  - `selenium_uc` 作为 stealth / challenge / gesture 专用路径
  - `playwright_cli` 作为轻量兼容路径
- 让文档、skill、默认调用建议更明确地引导使用方优先走最强主路径
- 尽量减少“普通 MCP 任务也必须手动切引擎才好用”的情况

### 3.6 高阶稳定性与边角收口

- 补齐复杂站点下仍然容易暴露的问题：
  - 结构化结果噪音过高
  - 某些局部交互后上下文断裂
  - 少数动作仍然需要多次探测
  - 页面状态漂移后的恢复建议不够直接
- 收紧复杂页面上的失败归因，让失败更容易区分：
  - 真失败
  - 页面未稳定
  - 目标漂移
  - 结构化数据不足
  - 需要切换引擎
- 收敛残余的浏览器操作边角，避免“主路径能跑，但少数真实场景还经常补刀”

### 3.7 诊断、可观测性与信噪比收口

- 继续压缩低价值诊断输出
- 让 `diagnose_page`、`action_trace`、`post_action_context` 之间的职责更清晰
- 对复杂动态页面，优先返回最有行动指导价值的信号，而不是堆大段原始内容
- 让调用方更容易判断：
  - 下一步是否应该直接继续
  - 是否应该切换引擎
  - 是否应该做更重的诊断

### 3.8 性能与空闲成本收口

- 在当前已完成的空闲轮询压降基础上，继续检查：
  - 高频动作后的隐式额外成本
  - 复杂页面结构化提取的耗时热点
  - GUI / daemon / worker 的稳态开销
- 对浏览器核心路径建立更稳定的性能基线
- 让“更强能力”尽量不以明显牺牲日常空闲体验为代价

### 3.9 文档、Skill、发布标准一次性收口

- README / 中文文档 / skill 模板统一对齐最终事实
- 把引擎分工、默认主路径、已知边界、推荐策略写清楚
- 把大版本验证与小版本 smoke 验证彻底固定下来
- 发布前验证、安装版验证、真实场景验证、cleanup 验证统一成固定标准

## 5. 测试计划

### 5.1 核心测试

继续扩展并运行以下方向的本地测试：

- `run_script` / `run_script_batch`
- `structured_page`
- 候选解析与排序
- `post_action_context`
- 动作 trace 连续性
- 三引擎 contract 一致性

### 5.2 浏览器核心验证

结合验证手册，重点增加这些验证项：

- 复杂动态页面上的结构化读取质量
- 高层动作的一次命中率
- 探测式 round-trip 是否减少
- 原始脚本探测依赖是否下降

### 5.3 发布验证

发布前仍然要做安装版验证，至少覆盖：

- GitHub 登录态真实流程
- 一个复杂动态已登录页面流程
- 一个 incognito 隔离流程
- 一个 challenge / gesture / stealth 场景
- 最终 cleanup 回到 idle

### 5.4 收口验证

在本计划末尾，追加一轮更偏产品收口的验证，至少覆盖：

- 复杂动态页面结构化读取连续性
- 默认 `patchright` 主路径的多步真实任务顺滑度
- 失败场景下的恢复建议是否足够直接
- 引擎切换建议是否清晰且必要
- 空闲 CPU / 稳态成本是否维持在可接受基线
- 文档与 skill 是否与最终行为一致

## 6. 验收标准

本计划只有在以下条件全部满足时才算完成：

- 代码实现完成
- README / 中文文档 / skill 模板与最终事实一致
- 相关测试通过
- 安装版发布验证通过
- 默认 `patchright` 主路径下的浏览器操作体验，明显比当前版本更接近官方 `playwright-mcp`
- 复杂页面上的结构化提取、高层动作、上下文连续性、失败恢复、引擎建议已经整体收口，不再需要再拆出新的主迭代
- 剩余问题只允许是极少量非阻塞边角，而不能再是主能力缺口

## 7. 非目标

本计划不做以下事情：

- GUI 大改版
- 引入第四个浏览器引擎
- 为 Gmail / YouTube Studio / GitHub 写死站点专用适配器
- 非必要地改动 profile 治理规则

## 8. 完成定义

这份计划的完成定义不是“做完一轮改动”，而是：

- 浏览器核心体验已进入接近 1.0 的状态
- 候选解析已经能够复用最近一次结构化页面上下文、交互区域、集合类型、工具栏/筛选/状态面标签，减少复杂页面 follow-up 时的盲扫与全页退化
- 默认主路径足够自然，普通任务不需要频繁人工切换策略
- 复杂页面上的主要体验短板已经解决
- 后续如果还有工作，应只剩零星维护，而不应再需要新的大主线计划
