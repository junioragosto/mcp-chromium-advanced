# 2026-06-19 统一产品化与性能收口计划

## 1. 目标

本轮不再做“修一个点、发一版、再补一个点”的零散迭代。

目标是把当前项目剩余的重要优化项一次性列全，并按一个长任务完成到可发布、可验证、可持续维护的状态。

本轮目标不是“再加几个功能”，而是把产品拉到下面这个标准：

1. GUI 轻量、稳定、状态一致
2. daemon / worker / GUI / MCP 之间职责清晰，互不打架
3. profile 占用、会话、外部进程、保活、脚本运行的治理口径统一
4. 默认使用体验接近官方 `playwright-mcp`，但保留三引擎分工
5. 空闲资源占用、频繁启停、窗口闪动、残留进程、状态漂移这些“产品级毛刺”尽量收干净
6. 文档、skill、发布包、启动方式、验证方式全部与当前事实一致

## 2. 当前问题总表

本轮统一处理的，不只是一两个 GUI bug，而是下面 10 类问题。

### 2.1 GUI 性能与刷新架构

当前最明显的结构性问题：

1. `refresh_table()` 仍然会整体重建 profile 表格
2. 每次刷新会重复创建行控件、按钮、下拉框、状态文本
3. 选中 profile 面板、底部状态栏、占用页、日志页存在重复查询
4. 非当前可见页面仍可能触发较重刷新
5. GUI 空闲时 CPU 仍明显偏高，不符合后台工具标准

### 2.2 状态一致性与单一真值源

虽然已做过多轮修正，但还没有彻底收口：

1. GUI 仍残留部分本地推断逻辑
2. control API 与 GUI fallback 之间仍有边界重叠
3. 手工打开浏览器、MCP 占用、保活运行、脚本运行的展示模型还不够统一
4. 某些极端切换下仍可能出现“实际已释放，GUI 还显示占用”或反过来的问题

### 2.3 daemon / worker / launcher 生命周期

当前需要彻底收口的点：

1. 根启动器、GUI、daemon、worker 的启动链路要绝对清晰
2. 退出、重启、异常退出、自启动恢复必须行为一致
3. 不能再出现“主 GUI 没了但 daemon/worker 还残留”或“只有部分组件起来”的情况
4. 启动失败、自恢复、端口冲突、孤儿进程都要有明确治理

### 2.4 占用治理与并发调度

当前系统已经支持多 profile 并行，但仍需进一步工业化：

1. profile 级锁、会话级锁、外部进程锁、保活锁、脚本锁需要统一口径
2. 同一 profile 的复用、拒绝、强制回收、超时回收、心跳失联处理要统一
3. “谁正在占用”要成为一等公民，而不是只显示空闲/使用中
4. GUI、control API、日志、事件流都要展示一致的 owner/source

### 2.5 引擎能力分层与默认体验

当前三引擎已存在，但产品层面还不够清楚：

1. `patchright` 应作为默认主引擎，承担最接近官方 `playwright-mcp` 的主路径
2. `selenium_uc` 作为反自动化增强与 challenge 场景引擎
3. `playwright_cli` 作为能力集成与轻量补充，而不是默认主路径
4. 上层调用要尽量不感知底层差异，但系统内部必须明确能力边界

### 2.6 与官方 `playwright-mcp` 的差距

这部分是后续产品成熟度的核心，不应再模糊处理：

1. 高层动作命中率仍有差距
2. 复杂动态页面的结构化提取能力仍有差距
3. 页面诊断、页面健康、失败恢复策略仍不够统一
4. 会话复用、标签稳定锚定、少探测执行这些方面仍需靠近官方体验

### 2.7 可观测性与诊断

当前日志和状态输出还不够产品化：

1. 事件、日志、占用、会话、进程之间缺少统一关联视角
2. GUI 日志页、控制台日志、MCP 日志还没有完全统一模型
3. 故障时定位信息还不够集中，用户很难第一时间知道问题在哪一层
4. 缺少更明确的轻量指标和运行健康快照

### 2.8 空闲资源占用与高频使用成本

用户真实反馈里，成本问题很明确：

1. 空闲 CPU 过高
2. 高频调用时命令行窗口一闪而过影响桌面
3. 浏览器窗口默认抢前台影响用户工作
4. 频繁一步一开一关的成本过高

### 2.9 发布、安装、打包、启动体验

当前安装包能力已基本成型，但还需要继续统一：

1. 根启动器是唯一入口
2. 自启动、手动启动、升级替换、退出行为必须统一
3. 打包结果、资源目录、skill 模板、说明文档都要跟代码事实一致
4. 发布验证要形成固定制度，而不是临时人工发挥

### 2.10 文档与 skill

当前最大风险之一不是代码，而是“文档和真实行为逐渐漂移”：

1. README、发布说明、安装说明、skill 模板、系统 skill 必须同步
2. 三引擎定位、默认引擎、如何切换、何时切换要写清楚
3. profile 标注账号与站点登录态不是一回事，这类高频误解要在文档里明确澄清

## 3. 本轮总原则

### 3.1 单一真值源

所有运行态真值必须以 `core / daemon control API` 为准。

GUI 不再负责拼真值，只负责：

1. 拉取快照
2. 消费事件
3. 展示状态
4. 触发操作

### 3.2 先收敛架构，再谈局部性能

本轮不接受只通过“调大 timer 间隔”来压 CPU。

要通过：

1. 渲染增量化
2. 查询分层
3. 事件驱动
4. 生命周期治理
5. 状态模型统一

去做真正的产品级优化。

### 3.3 三引擎不是完全同级

保留统一入口，但内部定位必须清晰：

1. `patchright`：主引擎，优先走它
2. `selenium_uc`：反检测/挑战页优先
3. `playwright_cli`：补充能力与集成通道

### 3.4 发布标准以真实验证为准

代码通过测试不等于可交付。

本轮以：

1. 本地测试
2. 安装态测试
3. 并发测试
4. 真实场景测试
5. 空闲性能测试

共同作为验收依据。

## 4. 统一工作流

本轮按 8 个工作流一次性收口。

## 5. 工作流 A：GUI 刷新架构重构

### 目标

把 GUI 从“全表重绘 + 高频重复查询”改成“快照拉取 + 增量更新 + 可见区域优先”。

### 实施项

1. 为 profile 表建立稳定的行级 view-model 快照
2. `refresh_table()` 改为 diff 更新，而不是全量重建
3. 行内控件复用，不重复 new widget
4. 选中 profile 面板与表格共享同一轮快照，避免重复查 control
5. 占用页、日志页、会话页在不可见时降频或暂停重刷新
6. GUI 定时刷新拆成：
   - 存活探测
   - 轻量状态刷新
   - 重详情刷新
   - 事件驱动局部刷新
7. 底部状态栏、概览面板、表格使用同一份 control snapshot

### 验收标准

1. 常规刷新不再整体重建表格
2. profile 数量较多时 GUI 不明显卡顿
3. 空闲 CPU 明显下降

## 6. 工作流 B：运行态真值收口

### 目标

彻底减少 GUI 本地状态推断，把 profile 占用、owner、会话、保活状态统一到 control 输出。

### 实施项

1. 梳理所有 GUI fallback 路径
2. 能迁到 daemon 的一律迁到 daemon
3. 定义统一 profile runtime state 结构
4. 明确 owner/source 枚举：
   - `idle`
   - `mcp`
   - `manual`
   - `keepalive`
   - `script`
   - `system`
5. 统一 action availability 计算逻辑
6. 清理 GUI 侧容易漂移的本地缓存
7. 外部 Chromium 进程识别结果进入统一 runtime snapshot

### 验收标准

1. GUI 不再显示与 control 矛盾的占用状态
2. 同一 profile 任一时刻只有一个对外主状态
3. 手工/MCP/脚本/保活切换时状态能在可接受时间内收敛

## 7. 工作流 C：事件驱动与查询分层

### 目标

减少“靠轮询碰运气刷新”的模式，改成“事件优先、轮询兜底”的控制面。

### 实施项

1. 补齐 profile runtime 相关事件
2. 为关键状态变化发事件：
   - acquire
   - release
   - manual_open
   - manual_close
   - keepalive_start
   - keepalive_stop
   - script_start
   - script_stop
   - reclaim
   - start_failed
3. GUI 优先消费事件做局部刷新
4. 轮询保留，但只作为兜底恢复机制
5. `/_control/status`、`/_control/profiles`、`/_control/events` 分轻重路径

### 验收标准

1. 主要状态切换不再只能靠轮询感知
2. GUI 对状态变化的感知延迟显著降低
3. control 路由不再被高频详情查询拖重

## 8. 工作流 D：launcher / daemon / worker 生命周期收口

### 目标

把“启动谁、退出谁、谁负责拉起谁、谁异常后谁恢复谁”彻底收清楚。

### 实施项

1. 固化根启动器为唯一用户入口
2. 明确 GUI 启动时 daemon/worker 的拉起策略
3. 明确 GUI 退出时 daemon/worker 的处理策略
4. 明确 `--start-minimized`、自启动、重启、替换升级的行为
5. 增加孤儿进程治理与日志
6. 增加端口占用、重复实例、自恢复失败时的明确提示
7. 统一安装态与源码态行为差异

### 验收标准

1. 用户只启动根 exe 即可正常带起整套系统
2. 自启动路径稳定
3. 不再出现“起了一半”或“退了一半”的混乱状态

## 9. 工作流 E：会话与并发治理加强

### 目标

把 profile 级并发治理进一步产品化，让“谁占用、为何占用、何时释放、怎样回收”完全明确。

### 实施项

1. 统一 profile 锁与 session 锁语义
2. 完善 session 心跳、超时、异常回收
3. 同一 profile 的复用与拒绝策略显式化
4. 外部脚本运行纳入同一治理模型
5. 占用状态栏显式展示 owner/source 与必要详情
6. 日志与事件里补 owner/source、session id、engine
7. 强化“释放后状态归位”的一致性校验

### 验收标准

1. 不会因并发场景轻易出现错乱占用
2. 强制回收和自然释放都能回到正确状态
3. 外部脚本、手工、MCP、保活四类场景都能共用同一治理模型

## 10. 工作流 F：引擎定位、默认体验与官方对齐

### 目标

让系统默认体验更接近官方 `playwright-mcp`，同时保留项目自己的三引擎优势。

### 实施项

1. 固化 `patchright` 为默认主引擎
2. 梳理高层动作优先走 `patchright`
3. `selenium_uc` 专注 challenge / 反自动化增强路径
4. `playwright_cli` 保持接入但不承担默认主路径
5. 补齐引擎能力矩阵与限制说明
6. 继续收敛高层动作、页面摘要、结构化提取、失败恢复、tab 稳定锚定
7. 参考官方 `playwright-mcp` 继续优化：
   - 少探测
   - 更稳定的高层动作
   - 更强的页面级诊断
   - 更明确的失败恢复路径

### 验收标准

1. 默认引擎使用体验明显优于 `playwright_cli` 主导时
2. 文档和 skill 能明确告诉使用方何时切换引擎
3. patchright 主路径的复杂页面稳定性继续提升

## 11. 工作流 G：可观测性与故障诊断

### 目标

把日志、事件、占用、会话、进程信息收敛成可诊断的统一视图。

### 实施项

1. 统一运行事件模型
2. 统一日志来源与字段
3. GUI 日志页优先基于 control API 消费
4. 增加轻量健康快照
5. 为异常恢复、失败重试、占用冲突增加明确日志
6. 为 GUI 状态异常、进程残留、守护失败提供直接诊断线索

### 验收标准

1. 看到异常时能快速定位是 GUI、daemon、worker、引擎、profile 占用哪一层的问题
2. 日志与 GUI 状态不再互相打架

## 12. 工作流 H：文档、skill、发布制度收口

### 目标

把“代码事实、发布包、说明文档、skill 行为约束”完全对齐。

### 实施项

1. 更新 README 面向用户的总说明
2. 更新安装、运行、发布、验证文档
3. 更新三引擎定位说明
4. 更新 profile 占用规则、并发规则、无痕规则说明
5. 更新 skill 模板
6. 更新系统 skill
7. 固化发布验证清单：
   - 编译
   - 替换
   - 启动
   - 空闲性能验证
   - 并发验证
   - 真实场景验证

### 验收标准

1. 文档和代码事实一致
2. skill 不再误导使用方
3. 发布时有固定验证流程，不靠临场发挥

## 13. 本轮不做的事

为了避免再次发散，本轮明确不做下面这些内容，除非它们直接阻塞上述目标：

1. 全新外部引擎栈替换当前三引擎体系
2. 远程 Web 管理台
3. 大规模 GUI 视觉重设计
4. 网站级业务适配逻辑扩张
5. 与具体业务网站深度耦合的规则

## 14. 测试计划

### 14.1 单元测试

覆盖：

1. GUI table/view-model diff 更新
2. profile runtime state 统一优先级
3. action availability 统一逻辑
4. event -> GUI 局部刷新链路
5. launcher/daemon/worker 生命周期判断
6. 外部 Chromium 识别与 owner/source 归类

### 14.2 集成测试

覆盖：

1. `/_control/status`
2. `/_control/profiles`
3. `/_control/events`
4. GUI 控制调用链
5. MCP acquire/release
6. keepalive start/stop
7. script acquire/release

### 14.3 安装态测试

覆盖：

1. 根 exe 启动
2. 自启动
3. 退出
4. 重启
5. daemon 恢复
6. worker 生命周期

### 14.4 空闲性能测试

基于真实安装态验证：

1. GUI 打开但不操作
2. GUI 最小化
3. daemon 空闲
4. 一次真实会话后回到 idle

记录：

1. GUI CPU
2. daemon CPU
3. worker CPU
4. control API 延迟

### 14.5 并发与治理测试

覆盖：

1. 多 profile MCP 并行
2. 手工打开 + 其他 profile MCP
3. 保活 + MCP
4. 脚本 + MCP
5. 释放、异常退出、超时回收

### 14.6 真实场景验证

按既有发布验证 skill 执行至少一轮完整复杂验证，包括：

1. 真实登录态复用
2. 至少一次并行场景
3. 至少一次无痕场景
4. 至少一次复杂后台页面场景

## 15. 验收标准

只有同时满足下面条件，本轮才算完成：

1. GUI 空闲资源占用明显下降，且不是通过牺牲状态一致性换来的
2. profile 占用状态在 GUI / control / 日志 上一致
3. 根启动器、自启动、退出、恢复链路稳定
4. `patchright` 默认主路径稳定
5. 并发与释放治理无明显残留错乱
6. 文档、skill、发布说明同步更新
7. 本地测试、安装态测试、真实场景验证通过

## 16. 执行顺序

本轮严格按下面顺序推进，不再来回插针：

1. 先完成 GUI 刷新架构与状态真值收口
2. 再完成事件驱动、查询分层、生命周期治理
3. 再完成并发治理、引擎默认路径与日志诊断补强
4. 最后统一做文档更新、打包、安装态验证、真实场景验证

## 17. 最终交付物

本轮最终应交付：

1. 完整代码实现
2. 测试通过结果
3. 安装态可运行版本
4. 更新后的 README / release / install / architecture / skill 文档
5. 一份验证记录，逐项对应本计划验收

## 18. 结论

后续不再按“单点补丁”方式推进。

这一版之后的工作方式应切换为：

1. 先按本计划把剩余产品化问题一次性收口
2. 再做统一测试与发布验证
3. 最终以可发布结果汇报，而不是以“下一轮/下一步”汇报
